나의 작업 원칙
Phase가 끝날 때마다 브라우저에서 직접 확인 가능한 체크리스트를 반드시 제공한다
커밋과 푸시는 사용자가 명시적으로 "커밋해줘"라고 말하기 전까지는 하지 않는다
새 기능이나 새 작업을 시작하기 전에는, 코드를 바로 짜지 말고 먼저 "인터랙션 체크리스트 + Acceptance Criteria(완료 기준)"를 질문 형태로 먼저 제시한다. 사용자가 답변한 후에 실제 구현을 시작한다.
## 새 UI 요소를 만들기 전에
- 새로운 화면 요소(버튼, 선택지, 카드 등)가 필요하면, 먼저 프로젝트 
  안에서 가장 비슷한 기존 요소를 찾는다 (예: 새로운 선택형 버튼이 
  필요하면 기존 필터 칩 스타일을 그대로 복제)
- 기존에 참고할 만한 게 전혀 없는, 정말 새로운 형태의 UI일 때만 
  새로 디자인하고, 이 경우엔 사용자에게 먼저 간단히 시안(설명 또는 
  스케치)을 보여주고 승인받은 후 진행한다
승인 요청 원칙

다음 세 가지 중 하나에 해당하면, 실행 전 반드시 한 줄로 설명하고 승인을 받는다:

되돌리기 어려운 작업 (삭제, 강제 push, DB 구조 변경 등)
외부에 공개되거나 남에게 영향을 주는 작업 (배포, PR 병합, 환경변수 변경 등)
비밀 정보(.env, API 키, 토큰)와 관련된 작업

단순 조회/읽기(git status, ls, cat 등)나 로컬 개발 서버 실행(npm run dev) 같은 안전하고 되돌리기 쉬운 작업은 설명 없이 바로 진행해도 된다.

작업 완료 전 영향도 확인 (3인 병렬 개발 필수 규칙)

지금 이 프로젝트는 팀원 3명이 각자 다른 기능을 동시에 개발 중이다. 팀원들은 코딩 경험이 없어서, 본인이 만든 변경이 다른 사람 화면/기능에 영향을 주는지 스스로 판단하지 못한다. 그래서 작업을 "완료"로 보고하기 전, 반드시 아래를 자체적으로 점검하고 사용자에게 알려준다.

이번 변경이 다른 화면이나 다른 기능에도 영향을 줄 수 있는지 확인한다
영향받는 부분이 있다면, 구체적으로 어떤 파일/화면인지 콕 집어서 알려준다
영향이 없는 경우에도 "다른 곳에는 영향 없음을 확인했다"고 짧게 언급한다
공통 컴포넌트, 공통 유틸 함수, 레이아웃 파일처럼 여러 기능이 같이 쓰는 파일을 건드릴 때는 특히 신중하게 확인하고, 이건 위 "승인 요청 원칙" 2번(남에게 영향을 주는 작업)에 해당하는 것으로 간주해 승인을 받는다
DB 스키마(schema.prisma) 변경 규칙
schema.prisma 파일 수정이나 npx prisma migrate dev 실행은 "되돌리기 어려운 작업"으로 간주해 반드시 사전 승인을 받는다 (승인 요청 원칙 1번 적용)
평소 개발 중(스키마 자체를 바꾸는 게 아닐 때)에는 npx prisma migrate dev를 실행하지 않고, npx prisma generate만 사용한다
스키마 변경이 필요한 상황이면, 사용자에게 "지금 다른 팀원이 스키마 작업 중인지 카톡방에서 먼저 확인해달라"고 요청한 뒤 진행한다
공통 값(상수) 재사용 원칙
카테고리(한식/중식/일식/양식/분식/카페/패스트푸드/기타), 구역(정문/상대/예대/ 후문/공대쪽문) 같은 프로젝트 공통 값은 새로 정의하지 않는다
새 기능을 만들 때 이런 값이 필요하면, 먼저 프로젝트 안에서 기존에 정의된 곳 (constants 관련 파일 등)을 찾아서 그대로 가져다 쓴다
기존 정의를 못 찾겠으면, 새로 만들지 말고 사용자에게 어디 있는지 먼저 물어본다
코드 스타일 통일
이 프로젝트는 Next.js(App Router) + TypeScript + Tailwind CSS v4 + shadcn(base-nova 스타일) 기반이다. 새 컴포넌트를 만들 때도 이미 쓰이고 있는 스타일링 방식과 폴더 구조를 따르고, 새로운 스타일링 방식이나 라이브러리를 임의로 추가하지 않는다
패키지를 새로 설치해야 하면, 설치 전에 사용자에게 알리고 설치한다 (package-lock.json이 팀 전체 기준이 되므로, 임의 설치 시 다른 팀원과 버전이 어긋날 수 있다)
