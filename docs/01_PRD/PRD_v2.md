# 학식 말고 뭐 먹지? PRD (v2 - 개발 상세화 버전)

# 1. 프로젝트 개요

**프로젝트명(앱명) :** 학식 말고 뭐 먹지?

**학식 말고 뭐 먹지?**는 대학생을 대상으로 학교 주변 식당 정보를 한곳에서 제공하는 반응형 웹 애플리케이션(Web Application)이다. 사용자는 식당 검색, 필터 기반 탐색, 메뉴 및 가격 조회, 리뷰 확인 기능을 이용하여 자신의 상황에 맞는 식당을 쉽게 찾을 수 있다. 또한 학생들이 직접 식당을 제보할 수 있어 데이터베이스를 지속적으로 확장할 수 있도록 설계한다.

# 2. 문제 정의

현재 학생들은 학교 주변 식당 정보를 네이버 지도, 카카오맵, SNS, 커뮤니티 등 여러 서비스에서 각각 찾아야 한다. 특히 학생들이 중요하게 생각하는 메뉴, 가격, 학생 리뷰, 구역 등의 정보를 한곳에서 확인하기 어렵다. 또한 캠퍼스가 넓어 단과대 위치별로 이용하는 식당이 달라지는데, 이를 반영한 서비스가 없다. 더불어 학생회에서 진행하는 식당 제휴이벤트 정보도 여러 채널에 흩어져 있어 학생들이 어떤 식당에서 무슨 혜택을 받을 수 있는지 파악하거나 기억하기 어렵다. **학식 말고 뭐 먹지?**는 이러한 정보를 통합 제공하여 학생들이 쉽고 빠르게 식당을 선택할 수 있도록 하는 것을 목표로 한다.

# 3. 프로젝트 목표

| 목표 | 내용 |
|---|---|
| 정보 통합 | 학교 주변 식당의 메뉴, 가격, 리뷰를 한곳에서 제공 |
| 검색 편의성 | 구역·카테고리·가격대 필터 조합으로 원하는 식당을 빠르게 탐색 |
| 학생 참여 | 리뷰 작성과 식당 제보 기능으로 사용자 참여 유도 |
| 운영 관리 | 관리자 페이지를 통해 식당 및 리뷰를 효율적으로 관리 |
| 차별화 (제휴이벤트) | 여기저기 흩어져 있는 학생회 제휴이벤트 정보를 식당별로 한곳에서 확인 가능하게 하여 타 서비스(네이버지도, 카카오맵 등)와 차별화 |

# 4. 타겟 사용자

**주요 타겟**
- 대학생
- 학교 주변에서 식사를 자주 하는 학생
- 메뉴와 가격을 미리 확인하고 싶은 사용자
- 학생 리뷰를 참고하여 식당을 선택하고 싶은 사용자
- 단과대/기숙사 등 본인 위치 근처 식당을 빠르게 찾고 싶은 사용자

# 6. 요구사항

## 6.1 기능 요구사항 (상세)

### 검색
- 사용자는 식당명 또는 메뉴명으로 식당을 검색할 수 있다.
- 검색은 Menu 테이블 전체(대표메뉴 여부와 무관하게 등록된 모든 메뉴)를 대상으로 한다.
- 실시간 검색(입력 중 자동완성)은 구현하지 않으며, 검색 버튼 클릭 또는 Enter 입력 시 결과를 표시한다.
- 검색 결과가 없을 경우 다음 문구를 표시한다: **"검색 결과가 없어요. 새로운 식당을 제보해보세요!"**

### 필터
- **구역**: 정문 / 상대 / 예대 / 후문 / 공대쪽문 (5개, 확정)
- **카테고리**: 한식 / 중식 / 일식 / 양식 / 분식 / 카페 (6개, "기타" 없음). 식당 하나는 카테고리를 1개만 가진다.
- **가격대**: 식당의 메뉴 중 최저가 기준으로 4구간(~5천원 / ~1만원 / ~2만원 / 2만원~)에 자동 분류한다.
- **제휴이벤트**: "제휴이벤트 중인 식당만 보기" 체크박스. 체크 시 아래 제휴이벤트 항목 기준으로 현재 진행 중인 식당만 노출한다.
- 네 필터는 각각 독립적으로 동작하며, 선택하지 않은 필터는 "전체"로 간주되어 조건에서 제외된다. (예: 구역만 선택 시 해당 구역의 모든 식당이 카테고리/가격/제휴여부 무관하게 노출)

### 정렬
- 정렬 옵션: 평점순(기본값) / 거리순 / 가격순(저렴한 순)
- **평점순**: 리뷰 평균 별점 기준 내림차순
- **거리순**: 사용자의 실시간 GPS 위치를 기준점으로 계산한 직선거리 기준. 위치 정보 접근을 허용하지 않은 사용자에게는 거리순 정렬 대신 학교 정문 고정 좌표 기준으로 대체 계산하거나, 정렬 옵션 선택 시 위치 권한 요청 안내를 표시한다.
- **가격순**: 메뉴 최저가 기준 오름차순

### 리뷰
- 별점은 5점 만점, 정수 단위(0.5점 단위 미지원)로 작성한다.
- 한 사용자는 같은 식당에 리뷰를 여러 개 작성할 수 있다.
- 리뷰 작성자 본인은 자신의 리뷰를 수정 및 삭제할 수 있다.
- 로그인한 사용자만 리뷰를 작성할 수 있다.

### 식당 제보
- 필수 입력값: 식당명, 위치(주소), 카테고리
- 선택 입력값: 메뉴, 가격
- 관리자는 제보 승인 시 사용자가 입력한 카테고리를 포함한 모든 값을 수정할 수 있다.
- 반려된 제보는 마이페이지 > 내 제보에서 상태(대기/승인/반려)를 확인할 수 있다.

### 제휴이벤트
- 학생회에서 진행하는 식당별 제휴이벤트 정보를 식당 상세 페이지에 노출한다. (예: "학생증 제시 시 아메리카노 500원 할인")
- 관리자는 제휴이벤트 등록 시 **시작일과 종료일**을 입력하며, 별도의 수동 on/off 없이 오늘 날짜가 해당 기간에 포함되는지 여부로 시스템이 자동으로 진행 여부를 판단한다.
  - 시작일 이전: 미노출
  - 시작일~종료일 사이: 배지 및 상세 내용 노출, 필터에도 포함
  - 종료일 경과: 자동으로 배지 및 필터 노출에서 제외 (별도 삭제 불필요, 데이터는 보존)
- 식당 목록 카드에는 제휴 진행 중인 식당에 한해 작은 "제휴" 배지를 표시한다.
- 식당 상세 페이지에는 배지와 함께 제휴이벤트 상세 내용(할인 혜택, 종료일 등)을 별도 영역에 노출한다.
- 필터 바에 "제휴이벤트 중인 식당만 보기" 체크박스를 제공하며, 위 자동 판별 로직과 동일한 기준을 사용한다.
- 이 기능은 여기저기 흩어져 있어 학생들이 찾거나 기억하기 어려운 학생회 제휴이벤트 정보를 한곳에서 확인할 수 있게 한다는 점에서 타 서비스 대비 **학식 말고 뭐 먹지?**의 핵심 차별점 중 하나이다.

### 즐겨찾기
- 즐겨찾기 개수 제한은 없다.
- 즐겨찾기 목록에서 식당명을 클릭하면 해당 식당 상세 페이지로 이동한다.

### 관리자
- 관리자 계정은 별도 가입 절차 없이, 일반 회원가입 후 관리자가 데이터베이스에서 해당 계정의 `role` 값을 `admin`으로 직접 변경하여 부여한다.
- 관리자는 식당, 리뷰, 회원, 제보, 제휴이벤트(시작일/종료일/상세내용)를 관리할 수 있다.

## 6.2 시스템 요구사항

- 모바일 환경에 최적화된 반응형 UI를 제공한다.
- 학교 주변 식당 데이터를 기반으로 서비스를 제공한다.
- Kakao 로컬 API(장소 검색)로 식당 데이터를 수집할 때, `category_name` 값에 "술집", "호프", "포차" 등 주점 관련 키워드가 포함된 장소는 수집 대상에서 제외한다. (서비스 목적상 주점 카테고리는 다루지 않기로 결정)
- 사용자 인증은 Neon(PostgreSQL) 기반 커스텀 인증(User 테이블에 이메일/비밀번호 저장 후 로그인 시 조회)을 사용한다.
- 지도는 Kakao Map API를 사용하며, 첫 화면(메인)에서 사용자의 실시간 위치를 지도 위에 표시한다.
- GitHub를 이용하여 버전 관리를 수행한다.
- 각 개발자는 독립된 Neon 프로젝트(브랜치)에서 작업하며, 컬럼명 등 스키마 규칙은 11번 ERD를 따른다.
- ORM은 Prisma를 사용하여 스키마를 코드로 관리하고, 이를 기반으로 Neon DB에 마이그레이션을 적용한다.
- 배포는 Vercel을 사용하며, Neon과의 연동을 통해 프로덕션 환경에서 서버리스 PostgreSQL을 사용한다.

## 6.3 제약사항

- 개발 기간은 2주이다.
- AI(Vibe Coding)를 활용하여 개발한다.
- 리뷰 사진, AI 추천, 인원수 기반 필터는 MVP 범위에서 제외한다.
- 서비스 대상은 특정 대학 주변 식당으로 한정한다.

# 7. User Flow & Admin Flow

## 7.1 User Flow

```
앱 실행
  │
  ▼
메인 화면 (지도 + 내 위치 표시)
  │
  ┌──────────────┴──────────────┐
  ▼                             ▼
식당 검색                 필터 선택 (구역/카테고리/가격대)
  │                             │
  └──────────────┬──────────────┘
                 ▼
         식당 목록 (정렬: 평점순/거리순/가격순)
                 │
                 ▼
              식당 상세
                 │
   ┌─────────────┼─────────────┐
   ▼             ▼             ▼
메뉴 확인      리뷰 확인      위치 확인
                 │
                 ▼
            리뷰 작성 버튼
                 │
            로그인 여부 확인
                 │
           ┌─────┴─────┐
           │  미로그인  │
           └─────┬─────┘
                 ▼
               로그인
                 │
                 ▼
           리뷰 작성 완료
                 │
                 ▼
              상세 페이지
```

---

**식당 제보**

```
메인 화면
  │
  ▼
식당 제보
  │
  ▼
식당 정보 입력
(식당명* / 위치* / 카테고리* / 메뉴 / 가격)
  │
  ▼
제보 등록
  │
  ▼
관리자 승인 대기
  │
  ▼
마이페이지 > 내 제보에서 상태 확인
(대기 / 승인 / 반려)
```

사용자는 메인 화면에서 식당을 검색하거나 구역·카테고리·가격대 필터를 조합하여 원하는 식당을 탐색할 수 있다. 필터는 하나만 선택해도 나머지는 전체로 간주되어 결과가 표시된다. 식당 상세 페이지에서 메뉴, 가격, 위치, 리뷰를 확인할 수 있으며, 리뷰 작성은 로그인한 사용자만 가능하다. 등록되지 않은 식당은 '식당 제보' 기능을 통해 관리자에게 등록을 요청할 수 있으며, 처리 상태는 마이페이지에서 확인 가능하다.

## 7.2 Admin Flow

```
관리자 로그인
  │
  ▼
관리자 대시보드
  │
  ┌─────────────┼─────────────┬─────────────┐
  ▼             ▼             ▼             ▼
회원 관리     식당 관리     리뷰 관리     식당 제보 관리
  │             │             │             │
  ▼             ▼             ▼             ▼
회원 조회     식당 추가     리뷰 조회     제보 목록 조회
회원 관리     식당 수정     리뷰 삭제     제보 내용 수정 가능
              식당 삭제                     (카테고리 등)
                                            │
                                        승인 / 반려
                                            │
                                            ▼
                                      승인 시 DB 등록
```

관리자는 관리자 대시보드를 통해 회원, 식당, 리뷰 및 식당 제보를 관리한다. 사용자가 등록을 요청한 식당은 관리자가 내용을 검토·수정한 뒤 승인하여 데이터베이스에 등록되며, 이를 통해 서비스의 식당 정보 품질을 유지·관리한다.

# 12. 필터·정렬 로직 참고 (개발자용)

각 개발자가 동일한 방식으로 구현하도록 쿼리 로직 예시를 공유한다. (Prisma ORM 기준)

```javascript
// 필터: 선택하지 않은 값은 조건에서 제외
const where = {}

if (zone !== '전체') where.zone = zone
if (category !== '전체') where.category = category
if (priceRange !== '전체') where.priceRange = priceRange

// 제휴이벤트: 오늘 날짜가 시작일~종료일 사이인 식당만 (체크박스 선택 시)
if (partnershipOnly) {
  const today = new Date()
  where.partnershipStartDate = { lte: today }
  where.partnershipEndDate = { gte: today }
}

// 정렬
let orderBy = {}
if (sortBy === '평점순') orderBy = { avgRating: 'desc' }
if (sortBy === '가격순') orderBy = { minPrice: 'asc' }
// 거리순은 DB 정렬이 아닌 애플리케이션 단에서 계산 후 정렬 (아래 참고)

const restaurants = await prisma.restaurant.findMany({ where, orderBy })
```

- 가격대 구간 분류는 `min_price` 값을 기준으로 프론트엔드 또는 서버 로직에서 4구간 중 하나로 매핑한다.
- 거리순 계산은 사용자의 실시간 GPS 좌표(`navigator.geolocation`)와 식당 좌표 간 직선거리(하버사인 공식)를 사용한다. 사용자 좌표는 매 요청 시 함께 전달되므로 DB 정렬이 아닌, 조회된 목록을 서버 또는 클라이언트에서 즉시 계산 후 정렬한다. 위치 권한이 거부된 경우 학교 정문 고정 좌표를 대체 기준으로 사용한다.
- 제휴이벤트 진행 여부는 별도 boolean 컬럼으로 관리하지 않고, 조회 시점에 `partnership_start_date`~`partnership_end_date` 구간에 오늘 날짜가 포함되는지로 매번 계산한다. 별도의 배치 작업이나 스케줄러 없이 항상 최신 상태가 반영된다.
- 식당 상세 페이지에서도 동일한 조건으로 배지 노출 여부를 판단하고, 조건을 만족하면 `partnership_info`를 함께 표시한다.
