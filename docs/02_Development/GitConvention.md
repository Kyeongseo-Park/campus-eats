# CampusEats 팀 협업 규칙 (Git Convention)

이 문서는 CampusEats 프로젝트를 함께 개발하는 팀원들이 일관된 방식으로 협업하기 위한 기본 규칙을 정리한 문서입니다.

## 1. 브랜치 전략

- `main` : 항상 정상 동작하는 안정 버전만 유지합니다. 직접 작업하지 않습니다.
- `feature/기능이름` : 새로운 기능을 개발할 때 사용합니다. (예: `feature/login`, `feature/search`, `feature/review`)
- `fix/버그이름` : 버그 수정 시 사용합니다. (예: `fix/login-error`)

**작업 흐름**

```bash
git checkout main
git pull                        # 최신 상태로 업데이트
git checkout -b feature/login   # 새 브랜치 생성 및 이동

# 작업...

git add .
git commit -m "feat: 로그인 UI 구현"
git push origin feature/login
```

브랜치 작업이 끝나면 GitHub에서 **Pull Request(PR)** 를 생성하여 `main`에 병합합니다.

## 2. 커밋 메시지 규칙

커밋 메시지는 `태그: 설명` 형식으로 작성합니다.

| 태그 | 의미 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 추가/수정 (PRD, README 등) |
| `style` | 코드 포맷팅, 세미콜론 등 (기능 변화 없음) |
| `refactor` | 코드 구조 개선 (기능 변화 없음) |
| `test` | 테스트 코드 추가/수정 |
| `chore` | 빌드 설정, 패키지 관리 등 기타 작업 |

**예시**
- `feat: 식당 검색 기능 구현`
- `fix: 리뷰 작성 시 별점 저장 안 되는 버그 수정`
- `docs: ERD 문서 업데이트`

## 3. Pull Request(PR) 규칙

- PR 제목은 커밋 메시지 규칙과 동일한 형식으로 작성합니다. (예: `feat: 식당 상세 페이지 구현`)
- PR 설명에는 아래 내용을 간단히 포함합니다.
  - 무엇을 구현/수정했는지
  - 테스트는 어떻게 했는지 (선택)
  - 스크린샷 (UI 변경이 있는 경우, 선택)
- 병합 전 팀원 1명 이상의 리뷰 승인을 받는 것을 권장합니다.
- 리뷰 후 `main`으로 병합(Merge)하고, 사용한 브랜치는 삭제합니다.

## 4. 충돌(Conflict) 예방

- 작업 시작 전 항상 `git pull`로 최신 상태를 받아옵니다.
- 같은 파일을 동시에 여러 명이 수정하지 않도록 사전에 담당 영역을 나눕니다.
- 충돌이 발생하면 당황하지 말고 팀원과 함께 어떤 코드를 남길지 상의 후 해결합니다.

## 5. 폴더 구조

```
docs/
├── 01_PRD          # 기획 문서 (PRD, 요구사항 정의 등)
├── 02_Development   # 개발 관련 문서 (API 명세, 설정 가이드 등)
├── 03_Design        # 디자인 관련 문서 (와이어프레임, 화면 설계 등)
└── 04_Meeting       # 회의록
```

## 6. 기타 규칙

- `.env` 파일, API 키 등 민감한 정보는 절대 커밋하지 않습니다. (`.gitignore`에 등록)
- 커밋은 작은 단위로 자주 하는 것을 권장합니다. (한 커밋에 너무 많은 변경사항을 담지 않기)
- 작업 중 애매한 부분은 혼자 판단하지 말고 팀 채널에 먼저 공유합니다.
